# Grok-Search 改造计划（思路文档）

> 这个文档不是给执行者看的步骤清单，是给理解者看的设计哲学。  
> 另一个 AI 看完应该理解"为什么这么改"，然后能自己设计方案落地。

---

## 一、当前问题（已通过实测验证）

### 问题 1：Grok 调研的"深度"有上限

**实测**：跑"5 个 LLM 推理框架对比"任务，单 Grok 4-agent 调用 73 秒，22 个来源，5 个框架里 TGI 只被提到 12 次（其他 15-23 次），**深度不均**。

**根因**：Grok 4.20 multi-agent-console 内部是 4 个 agent 并行 + 1 个 captain 合成。即使 prompt 强制让 Grok 分两 section 输出，实测还是 2 倍失衡（Cursor 2492 字 vs Claude Code 4862 字）。

**业界参考**：Anthropic 多 agent 论文（https://www.anthropic.com/engineering/multi-agent-research-system ）指出，multi-agent 在"广度优先"任务上比单 agent 提升 90.2%，代价是 4-15× token。但**他们的 multi-agent 是 1 个 lead + N 个 subagent，每个 subagent 有独立 context window**。

**关键问题**：我们的 deep 模式是"1 个 Grok 调用 + 2 个补源 API"，**没有任何 multi-agent 编排**。这导致：
- 5 对象对比的覆盖度严重不均
- 复杂任务（5×5=25 个交叉点）无法结构化输出
- 调用方拿到的"看似完整"的答案可能漏掉关键信息

### 问题 2：当前架构在"浪费"和"不够"之间反复横跳

**实测**：
- 简单 deep 任务（"X 是什么"）也跑 Brave + Intent 补源——3-5s 浪费
- 复杂 deep 任务（5 对象 5 维度）单 agent 又不够——1-2 个对象被忽略

**根因**：deep 模式是"固定配方"（1 Grok + Brave + Intent），没有按任务复杂度分级。

**用户原话**（你的判断）："大部分时候单 agent 就够，部分时候 2 个 grok agent，少数时候 3 grok agent。简单 deep 不需要 3 路并行。"

**业界参考**：Anthropic scaling rules——"Simple fact-finding requires just 1 agent with 3-10 tool calls, direct comparisons might need 2-4 subagents with 10-15 calls each, complex research might use more than 10 subagents"。

**关键问题**：当前没有"档位"概念，所有 deep 任务一视同仁。

### 问题 3：Grok 答案的"准确性"被严重低估

**实测（R4 案例）**：用户问"Pi 怎么搞"（一个简称），Grok 自信地给出了 1877 字关于"Pi Network 加密货币"的内容——**完全答错了用户的问题**（用户想的是 Pi Coding Agent AI 工具）。这个回答看起来完整、有细节、有来源（5 个），但**核心答案是错的**。

**我之前的评估偏差**：我只看了"完整性"（覆盖度、结构、长度、来源数），**完全没看"准确性"**。这是一个根本性错误。

**关键洞察（你的判断）**：
> "这个项目的核心逻辑是，让 grok 输出准确的回答的同时，最大的利用他背后强大的搜索资源，最后把 grok 的回答和搜索资源+api 的补源，一起给到你，让你做出判断和结构性的回答。直接给到 grok，他的回答看上去很好，但是是不是对的，你能不能判断。"

**核心设计哲学（必须理解）**：
- grok-search 不是"问答系统"，是**原料聚合器**
- 输出的 `content` 字段是 **Grok 整合后的内容（参考，不是答案）**
- 调用方 AI 拿到原料后**自己判断真伪、重新组织**
- 这意味着 grok-search 必须**暴露更多原料**，不只是返回 Grok 加工后的内容

### 问题 4：碎片化输入 vs AI 改写的真实价值

**实测**：
- R1（"x 是啥"）：裸 prompt 让 Grok 正确反问 + 答对，AI 改写强行解读为 Pi Coding Agent 反而错
- R2（"a 和 b 哪个好"）：裸 prompt 让 Grok 正确反问"a/b 是什么"，AI 改写强行解读为 Cursor vs Claude Code 反而错
- R3（"国内大模型选型"）：两者差不多
- R4（"Pi 怎么搞"）：裸 prompt 误判为 Pi Network 加密货币，AI 改写加了 context 才答对

**关键洞察**：
- 碎片化输入下，**Grok 的反问能力是优点**，AI 改写经常扼杀它
- AI 改写的价值是"加 context 消除歧义"，不是"替代用户原话"
- 真实用户输入经常是碎片化的，**不要自动改写**——但**要让 Grok 知道怎么反问**（通过 prompt 模板）

### 问题 5：我之前的"质量评估"完全偏了

**我之前评判的维度**：完整性、结构化、长度、来源数  
**我漏掉的维度**：准确性、可验证性、幻觉率、歧义处理

**核心错误**：我把"看起来完整"等同于"质量好"，这导致：
- R4 案例的 Pi Network 回答会被我打高分——但完全答错
- Q2 (Cursor vs Claude Code) 裸 prompt 30s 的回答反而比 AI 改写 63s 的"更真实"——因为它来自真实用户视角

**修复方向**：评估标准必须把"准确性"放在"完整性"之上。

## 二、改造的设计哲学（核心思路）

### 哲学 1：调用方责任 vs grok-search 责任

| 角色 | 责任 |
|------|------|
| **调用方 AI** | 决定要不要搜索、要不要拆任务、怎么拆、怎么合成、怎么验证 Grok 答案 |
| **grok-search** | 执行调研、暴露原料、**不合成**、**不自动改写**、**不做最终判断** |

**为什么**：调用方知道自己要什么格式、什么场景、什么约束。grok-search 是工具，不是代理决策者。

**反例**：synthesizer subagent 让 Grok 合成——这是过度设计，因为 Grok 不知道调用方想要什么格式。

### 哲学 2：3 档决策树（按任务复杂度分级）

| 档位 | 适用 | 调研方式 | 补源 | Token 成本 |
|------|------|---------|------|----------|
| **0 - Simple** | "X 是什么"、定义、概念 | fetch 1 个权威源 | 不上 | 极低 |
| **1 - Single Agent** | 单产品/单主题、一致性调研 | 1 个 Grok 调用 | 可选 | 1× |
| **2 - Multi-Agent** | 多对象对比、跨领域、5+ 对象 | N 个 Grok 并行 | lead agent 自己做 | 2-8× |

**为什么不是"按 N 个 agent 算"**：一致性主题（如"国内 LLM 选型"）用 1 个 xhigh（16 agent 内部已并行）可能比 5 个 subagent 更好。**对象/方向是否独立可分**才是判断标准，不是"任务看起来复杂就要 multi"。

**反例**：把"国内 LLM 选型"拆成 5 个 subagent 调研每个模型——这是浪费，因为有"统一性"主题用 1 个 xhigh 内部已经并行调研了。

### 哲学 3：调用方必须显式指定 multi-agent subagent 类型

**两种 subagent 类型**：

| type | 用途 | 是否依赖其他 subagent |
|------|------|:--:|
| `subject` | 独立调研单个对象 | 否 |
| `comparison` | 找多个对象的差异点 | 是（依赖 subject 输出） |

**为什么必须显式**：默认值会让调用方懒得思考，导致误用。
- 不指定 type → 报错
- type=subject → 必须有 subject 字段
- type=comparison → 必须有至少 2 个 subjects

**为什么需要 comparison subagent（关键洞察）**：
- 你提的"对比 A vs B"问题——subject 调研 A 和 subject 调研 B 各自只能得到"自己方向"的答案，**对另一方毫无所知**
- 真正的"差异点"需要 1 个 comparison subagent 专门看"两个都提到"的差异
- 没有 comparison subagent，调用方拿到的只是"A 的描述"和"B 的描述"，没有差异表

**反例**：直接让 Grok 调研"A vs B"——会偏向某个对象，或者写出虚假的对比表。

### 哲学 4：不合成（不自动整合）

**核心**：multi-search 返回 N 份独立结果，**不合成**。调用方自己整合。

**为什么**：
- 不同的调用方需要不同的合成方式（对比表 / 推荐列表 / 总结报告）
- Grok 不知道调用方的格式需求
- 合成是"创作"，不是"调研"——属于调用方的责任

**反例**：让 Grok 做 synthesizer——会出现"格式不对""遗漏关键点""重复内容"等问题。

### 哲学 5：暴露原料，不只给"加工后的内容"

**当前返回**：
```json
{
  "content": "Grok 整合后的回答",
  "primary_sources": [{"url": "..."}]  // 只有 URL
}
```

**应该返回**：
```json
{
  "content": "Grok 整合后的回答（参考）",
  "grok_sources": {
    "items": [
      {
        "url": "...",
        "fetched_content": "...（实际抓取的内容，调用方可以验证）..."
      }
    ]
  },
  "supplement_sources": {
    "brave": {...原始结果...},
    "serper": {...原始结果...}
  },
  "verification": {
    "verified_claims": [...],
    "hallucination_risk": "low|medium|high"
  }
}
```

**为什么**：
- 调用方要能验证 Grok 的关键论断（R4 案例就是验证失败）
- fetched_content 是"原料"，让调用方自己比对
- verification 是"辅助"，不是"保证"

**关键限制**：
- fetched_content 限制 5000 字/URL——避免响应过大
- 最多抓 5 个 URL——避免超时
- verification 用简单正则提取数字——**不是真正 NLU**，是参考

**反例**：让 grok-search 做"完全准确性保证"——做不到，因为 Grok 本身可能幻觉。

### 哲学 6：不自动改写用户原话

**核心**：用户原话直接传给 Grok，不做 AI 改写。

**为什么**：
- 用户的"暗含约束"（如"挑一个推荐"、"做企业应用"）比 AI 改写的硬约束更有效
- 用户的"模糊性"（如"a 和 b"）是 Grok 反问的触发器
- AI 改写经常"加太多"或"加错方向"

**但是**：Grok prompt 模板里要有"如何反问"的引导——
> "如果用户问题模糊（无具体对象、缺少时间/场景约束），请反问 1-2 个澄清问题再回答。"

**反例**：AI 改写把"Pi 怎么搞"改成"调研 Pi Coding Agent 2026 年功能"——这是猜测用户意图，可能猜错。

### 哲学 7：Grok prompt 模板升级（去硬约束，加自适应）

**当前**（4 句）：
```
你是资深 X，可自主使用 web_search 等工具调研。
你的任务是：Y。
每个事实标注来源 URL。
请用中文输出。
```

**改进**（10 句）：
```
你是资深 {role}，可自主使用 web_search、X/Twitter 搜索等工具进行独立调研。
你的任务是：{task}。
【研究纪律】
1. Start wide, then narrow down：先用 2-3 个宽泛 query 摸清信息全貌，再针对关键信息用具体 query 深入。
2. 每个事实性断言标注来源 URL。
3. 不要重复搜索同一关键词。
4. 充分利用 X/Twitter 独家数据源（开发者真实反馈）。
【来源偏好】
- 优先级：官方文档 > 独立 benchmark 评测 > 开发者社区反馈 > 营销博客
- 同一事实至少 2 个独立来源交叉验证
请用中文输出，篇幅根据任务复杂度自适应。
```

**关键变化**：
- 去"4000-6000 字"硬约束（实测：EASY 任务 IMPR 多出 6000 字都是用户没问的内容）
- 加"自适应篇幅"（让 Grok 自己决定）
- 加"研究纪律"（start wide、来源偏好、避免重复）
- 加 X/Twitter 提醒（用独家数据源）

**为什么硬约束有害**：
- EASY 任务写 5000 字需要凑内容（市场背景、竞品对比——用户没问）
- HARD 任务写 5000 字不够（5×5=25 个交叉点）

**实测对比**：
- 改进后来源数 +50%
- 5 维度覆盖度 5×2 倍提升
- 耗时 +30%（值得）

### 哲学 8：分级补源（简单 deep 不跑 3 路）

**核心**：补源按档位分级，不是"凡是 deep 都跑 3 路"。

| 档位 | 补源策略 | 理由 |
|------|---------|------|
| 0 - Simple | 不跑补源 | fetch 1 个权威源就够 |
| 1 - Single | 可选补源（`--no-supplement` 跳过） | 简单 deep 任务 Brave 1-2s 价值低 |
| 2 - Multi | 只在 lead agent 跑补源 | subagent 自己决定要不要 |

**为什么**：
- 简单 deep 跑 Brave 是浪费（5-10s）
- 多 agent 模式下，每个 subagent 跑补源 = N 次补源（浪费）
- 应该让 lead agent 做总补源，subagent 用 web_search 工具

**反例**：所有 deep 模式无脑跑 3 路——慢、浪费 token、补源结果质量参差。

### 哲学 9：迭代深度（只跑 1 轮，调用方决定要不要深挖）

**核心**：grok-search 不做"自动迭代"，只跑 1 轮。

**为什么**：
- 不同调用方对"深度够不够"的判断标准不同
- 迭代方向取决于上一轮结果——这需要调用方看结果再决定
- 自动迭代容易陷入"无限深挖"循环

**模式**：
```
调用方跑第 1 轮 → 看结果 → 决定要不要跑第 2 轮
                ↓
         写新 prompt（针对性补强）→ 跑第 2 轮
```

**反例**：自动跑 2 轮（不管结果满不满意）——浪费 token、可能跑错方向。

### 哲学 10：Grok 4.20 内部多 agent 架构要理解清楚

**关键事实**（已验证）：

| 模型 | 内部 agent 数 | 用途 |
|------|:--:|------|
| `grok-4.20-multi-agent-console` | 4 agents | 默认 |
| `grok-4.20-multi-agent-high` | 4 agents（高 effort） | 中等 |
| `grok-4.20-multi-agent-xhigh` | 16 agents | Heavy |

**关键认知**：
- 4/16 agent 是**模型内部能力**，客户端不需要自己拆
- "multi-agent 模式"在客户端 = **多次独立 Grok 调用**（每次调用内部 4/16 agent 并行）
- 选模型 = 选每次调用的"内部并行度"
- console vs xhigh 差距：实测 xhigh 慢 50%，质量略好

**反例**：客户端自己实现 4/16 agent 编排——重复造轮子，且 Grok 模型升级时容易失效。

## 三、5 个改造方向（设计意图）

### 方向 1：暴露原料给调用方

**意图**：让调用方能验证 Grok 的关键论断。

**具体设计**：
- `grok_sources[i].fetched_content`：Grok 引用的 URL 实际抓取内容
- `supplement_sources`：补源 API 原始结果（不是过滤后的）
- `verification`（可选，--verify 参数）：关键事实交叉验证

**为什么这个重要**：R4 案例证明，Grok 输出"看起来好"≠ 正确。调用方必须能验证。

**权衡**：
- fetched_content 限制 5000 字/URL（避免响应过大）
- 最多抓 5 个 URL（避免超时）
- 不抓"重要 URL"（不做优先级判断）——按 primary_sources 顺序

### 方向 2：multi-agent 模式

**意图**：解决"单 agent 调研有覆盖度上限"的问题。

**具体设计**：
- 新增 `multi-search` 子命令
- 接受 `--tasks-json` 参数（每个 task 显式指定 type）
- subject 任务并行，comparison 任务依赖 subject
- **不合成**，返回 N 份独立结果

**为什么需要 comparison subagent**：
- subject 调研 A 和 subject 调研 B 都"对另一方毫无所知"
- 只有 comparison 才能找到"A 和 B 的差异点"
- 没有 comparison，调用方拿到的只是"A 的描述"+"B 的描述"，没有"差异表"

**反模式**：
- ❌ 默认 type（让调用方懒于思考）
- ❌ synthesizer（让 Grok 合成）
- ❌ 自动拆（让 Grok 决定怎么拆任务）

### 方向 3：分级补源

**意图**：简单 deep 不浪费，复杂 deep 才补源。

**具体设计**：
- 新增 `--no-supplement` 参数（跳过 Brave + Intent）
- 默认行为保持兼容（继续 3 路）
- 简单 deep 任务调用方显式加 `--no-supplement`

**为什么不默认跳**：保持向后兼容，避免破坏现有调用方。

**反模式**：
- ❌ "凡是 deep 都跑 3 路"——浪费
- ❌ "凡是 deep 都不跑补源"——损失信息

### 方向 4：准确性辅助

**意图**：帮助调用方快速识别 Grok 答案的"风险等级"。

**具体设计**：
- `--verify` 参数触发
- 自动提取 Grok 答案中的关键数字/版本号/价格
- fetch 原文做简单比对
- 输出 `hallucination_risk: low/medium/high`

**权衡**：
- 用简单正则提取（不是真正 NLU）——会有误判
- 作为"参考"，不是"保证"
- 调用方拿到后自己判断

**反模式**：
- ❌ "verifier 保证准确性"——做不到
- ❌ "完全跳过验证"——失去价值

### 方向 5：Grok prompt 模板升级

**意图**：让 Grok 的回答更准确、覆盖更广，但不过度凑字数。

**具体设计**：
- 模板从 4 句升级到 10 句
- 加"研究纪律"（start wide、来源偏好、避免重复）
- 去"4000-6000 字"硬约束
- 加"自适应篇幅"

**实测效果**：
- 来源数 +50%
- 5 维度覆盖度 +50-100%
- 耗时 +30%

**反模式**：
- ❌ "字越多越好"——实测 EASY 任务多出 6000 字是用户没问的内容
- ❌ "完全不给模板"——失去纪律

## 四、不改的东西（明确放弃）

| 不改 | 原因 |
|------|------|
| **AI 自动改写用户原话** | Grok 反问能力是优点，AI 改写经常扼杀它（R1/R2 案例） |
| **`disable-model-invocation: true`** | 用户希望模型自动判断触发 |
| **模型缓存 TTL** | 用户明确不要跨会话共享 |
| **移除 News API** | 保留选项，仅真正新闻事件用 |
| **synthesizer subagent** | 集成是调用方的事，不是 Grok 的事 |
| **自动迭代深度** | 调用方自己决定要不要深挖 |
| **改 `format_markdown()` 输出** | 保持兼容 |

## 五、关键决策的"为什么"

### 为什么是 3 档不是 N 档？

- 0 档（不上 Grok）和 1 档（单 agent）的工具完全不一样
- 2 档（multi-agent）是"调用方负责拆"的独立范式
- 3 档足够覆盖所有场景，再多就是 over-engineering

### 为什么 multi-agent 不默认开？

- 一致性主题用 1 个 xhigh（16 agent 内部已并行）更好
- 多对象对比才需要拆
- 默认开 = 浪费 token + 响应时间变长

### 为什么 subagent type 必须显式？

- 默认值会让调用方懒于思考
- 误用 subject/comparison 会导致结果错
- 显式 = 强制调用方想清楚

### 为什么 fetched_content 限制 5000 字？

- 不限制：响应可能几十 MB，调用方解析慢
- 5000 字：足够调用方验证关键事实，不至于过大
- 调用方如需全文，自己 fetch

### 为什么 verification 用简单正则？

- 真正 NLU 提取 claim 需要再调 LLM——慢、贵
- 简单正则已经能抓"含数字/版本号/价格"的关键事实
- 作为参考，不作为保证

### 为什么 max_fetch=5？

- 5 个 URL 抓取 = 5-15 秒
- 太多：超时
- 太少：验证不充分
- 5 是平衡点

### 为什么 3 档决策树按"任务复杂度"而不是"对象数"？

- "5 个对象对比"用 1 个 xhigh 内部已经并行调研
- "2 个对象对比"也用 1 个 xhigh 可能就够
- 真正需要 multi-agent 的是"对象**独立性**高"的任务
- "X 行业 Y 和 Z 趋势"——Y 和 Z 独立，可以拆
- "国内 LLM 选型"——5 个模型是同一主题，1 个 xhigh 内部并行更连贯

## 六、验收标准（行为正确性）

### 行为正确

- ✅ 简单查询（"X 是什么"）走 fetch，不上 Grok
- ✅ 单主题调研走 1 个 Grok 调用
- ✅ 多对象对比走 multi-search（subject + comparison）
- ✅ Grok 答案中引用的事实能在 `grok_sources[i].fetched_content` 里找到原文
- ✅ 调用方拿到 N 份独立结果，自己整合
- ✅ 调用方能识别 Grok 答案的 hallucination_risk

### 性能影响

- ✅ 简单 deep（--no-supplement）：不增加耗时
- ✅ 普通 deep：增加 5-15s（抓 5 个 URL）
- ✅ --verify deep：增加 10-30s（抓 + 验证）
- ✅ multi-search 2 subject：~90s 并行（同 single agent）
- ✅ multi-search 2 subject + 1 comparison：~150s 串行

### 兼容性

- ✅ 不删除/重命名现有字段
- ✅ 不改现有 CLI 命令的行为
- ✅ 不改 `_call_grok_search()` 内部
- ✅ 不改 smart_search 模块
- ✅ SKILL.md 描述与实际行为一致

## 七、让 AI 理解的关键问题（"为什么"清单）

新 AI 在动手前必须理解：

1. **为什么是 3 档**？——按"工具 + 调用方责任"分，不是按"对象数"
2. **为什么 multi-agent 不默认**？——一致性主题用单 xhigh 内部并行更连贯
3. **为什么 subject / comparison 必须显式**？——默认值会鼓励误用
4. **为什么 fetched_content 限制 5000 字**？——平衡响应大小和验证需求
5. **为什么 verification 用简单正则**？——真正 NLU 太贵，简单正则已够用
6. **为什么不合成**？——不同调用方需要不同格式，集成是调用方责任
7. **为什么不自动改写**？——Grok 反问是优点，AI 改写经常扼杀它
8. **为什么 Grok 4.20 4/16 agent 不需要客户端实现**？——那是模型内部能力
9. **为什么 max_fetch=5**？——平衡超时风险和验证充分性
10. **为什么"准确性"比"完整性"重要**？——R4 案例证明完整 ≠ 正确
